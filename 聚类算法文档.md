# 舆镜（YuJing）语义事件聚类算法文档

## 一、算法概述

舆镜系统的核心任务是将来自多个平台（微博、百度、头条、知乎、B站、澎湃、华尔街见闻、财联社）的热点文章，自动聚合为"事件"。

算法流水线为：

```
文章标题/摘要 → Sentence-BERT 向量化 → FAISS 近邻检索
→ 双层 Otsu 自适应阈值 → Union-Find 聚类 → 簇内复核
→ 代表文章选择 + 重要性评分 → 事件元数据生成
```

**不使用任何硬编码阈值**，所有关键参数均由数据分布自适应决定。

---

## 二、向量化（Embedding）

| 配置项 | 值 |
|--------|-----|
| 模型 | `BAAI/bge-small-zh-v1.5` |
| 维度 | 512 |
| 池化 | Mean Pooling（attention mask 加权平均） |
| 归一化 | L2 归一化（向量模长为 1） |
| 最大长度 | 128 tokens |
| 批大小 | 16 |

**编码输入**：`文章标题 + 摘要/描述`，截断到 128 tokens。

由于向量已做 L2 归一化，两个向量的点积（`np.dot(va, vb)`）等价于余弦相似度，取值范围 `[-1, 1]`。

向量存储在 SQLite 的 `article_embeddings` 表中，以 BLOB 格式保存原始 float32 字节，避免重复推理。

---

## 三、近邻检索（FAISS）

使用 FAISS 的 `IVF-Flat` 索引加速大规模向量检索。

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `nlist` | 64 | 倒排列表数量（聚类中心数） |
| `nprobe` | 8 | 搜索时扫描的列表数 |
| `top-K` | 20 | 每篇文章检索的最近邻数 |

**流程**：

1. 将所有文章向量堆叠为矩阵 `(N, 512)`，训练 IVF-Flat 索引
2. 对每篇文章搜索 top-K=20 最近邻
3. 收集**全量** cosine 相似度，不做任何硬门槛过滤
4. 构建 `raw_pair_scores` 字典：每对文章取最大 cosine 值（双向取大）

这一步输出约 N × 20 / 2 个去重 pair 及其 cosine 分数。

---

## 四、双层 Otsu 自适应阈值

这是算法的核心创新：利用 Otsu 方法（最大类间方差法）从数据分布中自动找到最优分割点，**完全不依赖手动调参**。

### 4.1 Otsu 方法原理

Otsu 方法在给定一组值和一个候选阈值 t 时，将数据分为两类（< t 和 ≥ t），计算类间方差：

```
σ²_between(t) = w₀ · w₁ · (μ₀ - μ₁)²
```

其中：
- w₀, w₁：两类占比
- μ₀, μ₁：两类均值

遍历所有候选阈值（步长 0.005），选择使类间方差最大的 t，即两类分得最"开"的点。

### 4.2 双层分割

对全量 top-K cosine 分数（通常 10~13 万对）执行**两层 Otsu 分割**：

#### 第一层（L1）：噪声过滤线

在 `[0.30, 0.75]` 区间搜索最优分割点，将全量 cosine 分为：

- **低组（噪声）**：随机相似度，无语义关联
- **高组（候选）**：存在一定语义关联的文章对

> 实测 L1 ≈ 0.565（58% 噪声 vs 42% 候选）

#### 第二层（L2）：合并判定线

在高组内部的 `[L1, 0.90]` 区间再次执行 Otsu 分割：

- **低子组（主题相似）**：同领域但不同事件（如两条不同的教育新闻）
- **高子组（同事件）**：描述同一事件的不同报道

> 实测 L2 ≈ 0.690（高组中 76% 主题相似 vs 24% 同事件）

### 4.3 阈值使用

| 阈值 | 用途 | 效果 |
|------|------|------|
| L1（过滤线） | 过滤 `raw_pair_scores`，只保留 cosine ≥ L1 的文章对 | 去除随机噪声，减少后续计算量 |
| L2（合并线） | Union-Find 聚类的合并条件 | 只有 cosine ≥ L2 的文章对才会被合并到同一簇 |

**回退机制**：当数据量不足 50 对时，使用固定回退值 `0.65`。

---

## 五、Union-Find 聚类

过滤后的文章对（cosine ≥ L1）中，所有满足 cosine ≥ L2（合并线）的文章对，通过 Union-Find（并查集）合并到同一连通分量。

```
对于每个文章对 (A, B)：
  如果 pair_scores[(A,B)] >= L2:
    union(A, B)
```

Union-Find 采用**路径压缩**优化，时间复杂度近似 O(α(n))。

最终，每个连通分量构成一个初始聚类簇。

---

## 六、簇内复核（Cluster Verification）

Union-Find 存在**链式传染**问题：如果 A≈B（cosine=0.72）且 B≈C（cosine=0.71），A 和 C 会被合并到同一簇，但 A 与 C 可能实际不相关（cosine=0.45）。

**复核机制**：

1. 选出簇的代表文章（见第七节）
2. 计算每个成员与代表的 cosine 相似度
3. 若 `cosine(member, representative) < L2`，将该成员踢出
4. 被踢出的文章各自成为独立的单篇事件

这一步有效解决了链式传染导致的"大杂烩"事件。

## 七、代表文章选择与重要性评分

### 7.1 综合评分公式

对簇内每篇文章计算：

```
importance = 0.6 × centrality + 0.4 × hotness_norm
```

**取值范围 0~1**，综合衡量文章在簇内的语义代表性和社会关注度。

#### 语义中心性（Centrality）

文章与簇内所有其他成员的**平均 cosine 相似度**：

```
centrality(a) = (1 / (|C| - 1)) × Σ cos(vₐ, vᵦ)    对所有 b ∈ C, b ≠ a
```

中心性越高，说明文章在语义空间中越"靠近"簇的中心，最能代表事件的核心含义。取值范围自然落在 `[L2, 1]`（因为簇内成员已经过 L2 阈值过滤）。

#### 归一化热度（Hotness Norm）

```
hotness_norm(a) = raw_score(a) / max(raw_score(a'))    对所有文章 a'
```

其中 `raw_score` 是文章的原始平台热度评分：

```
raw_score = max(0, 120 - rank) + min(hot_value / 10000, 160) + 20 × I[source ∈ {微博, 知乎}]
```

- **榜单排名分**（0~120）：排名越靠前分越高
- **平台热度分**（0~160）：从 `extra_info` 提取原始热度值，线性映射
- **来源加权**（0/20）：微博、知乎额外加分（用户讨论度高的平台）

注意：`raw_score` 仅作为 importance 公式中 40% 权重的输入分量，由全局最大值归一化到 `[0, 1]`。**聚类决策完全由 cosine 驱动，不受热度影响。**

### 7.2 代表选择

**importance 最高**的文章成为该事件的代表文章（`representative_article`），用于：
- 事件标题（取代表文章标题）
- AI 摘要（对代表文章做 LLM 总结）
- 事件卡片封面展示

### 7.3 重要性分数存储

每篇文章的 importance 分数在聚类阶段预计算，存储在 `EventArticle.importance_score` 字段中（范围 0~1），供前端"按相关性排序"使用，无运行时开销。

## 八、事件元数据生成

每个聚类簇对应一个事件（Event），元数据构建规则：

| 字段 | 计算方式 |
|------|----------|
| `title` | 代表文章的标题 |
| `summary` | 聚合摘要（多篇文章关键信息合并） |
| `keywords` | TF-IDF 提取 + 去停用词 + 去重叠 |
| `sentiment` | 多数投票（文章情感标签众数） |
| `article_count` | 簇内文章数 |
| `platform_count` | 覆盖的不同平台数 |
| `heat_score` | 事件级热度（见下方公式） |
| `latest_article_time` | 最新文章的抓取时间 |
| `representative_article_id` | 代表文章 ID |

### 事件级热度公式（heat_score）

用于主页事件列表排序，衡量事件整体热度：

```
heat_score = avg(top-3 raw_scores) + min((platforms - 1) × 30, 150) + log₂(count) × 15
```

- **头部文章热度均值**：取 raw_score 最高的 3 篇取平均，避免长尾噪声拉高
- **跨平台覆盖奖励**：每多覆盖一个平台 +30，上限 150
- **报道数量奖励**：对数增长，避免数量膨胀主导

---

## 九、辅助相似度（Composite Similarity）

除了纯 cosine 用于聚类决策外，系统还计算一个多维组合相似度用于 `EventArticle.relation_score`（存储用途，**不参与聚类**）：

```
composite = 0.60 × cosine + 0.25 × time_decay + 0.15 × platform_diff
```

| 分量 | 权重 | 说明 |
|------|------|------|
| cosine 相似度 | 0.60 | 语义向量点积 |
| 时间衰减 | 0.25 | 发布时间差的指数衰减，半衰期 12 小时 |
| 跨平台奖励 | 0.15 | 不同平台 = 1.0，同平台 = 0.0 |

## 十一、关键设计决策

| 决策 | 理由 |
|------|------|
| 纯 cosine 做聚类决策 | composite（含时间/平台权重）会让语义不相关但时间接近的文章被错误合并 |
| 双层 Otsu 而非固定阈值 | 数据分布随时间变化（突发事件 vs 日常），自适应阈值更鲁棒 |
| Union-Find + 簇内复核 | Union-Find 高效但有链式传染，复核兜底解决 |
| centrality 0.6 + hotness 0.4 | 代表文章应语义最居中（能代表事件）且有足够关注度；centrality 占主导确保语义代表性 |
| importance_score 预计算 | 避免前端打开事件详情时实时做 O(n²) 向量运算 |
| composite 仅存储不聚类 | 保留多维信息供前端展示和诊断，但不干扰聚类精度 |
| raw_score 仅作为 importance 子分量 | 平台热度跨源量级差异大，不适合单独做排序依据，但归一化后作为 40% 权重输入是合理的 |
