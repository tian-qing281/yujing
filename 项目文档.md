# 舆镜 YuJing 项目说明文档

## 1. 项目定位

`yujing` 是一款全平台舆情监控与 AI 研判系统（舆镜 YuJing）。系统通过对多平台热榜的实时采集、语义化聚类与 AI 增强分析，建立起从"海量噪声"到"决策结论"的处理链条。

### 核心能力体系：
-   **信号层 (Article)**: 实时监控全网 8 个核心权重平台的热榜数据。
-   **事件层 (Event)**: 利用 TF-IDF + 归一化标题聚类算法，将碎片信号收敛为事件单元。
-   **研判层 (AI Analysis)**: 基于 SSE 流式架构，驱动 DeepSeek LLM 结合实时内容输出深度分析报告。
-   **智能体层 (Tool-Calling Agent)**: 9 个舆情原子能力以 OpenAI Function Spec 注册，DeepSeek V3 自主规划 2-5 步调用链给出带 `event#N` / `article#N` 可溯源引用的结构化研判；20 query 评测 100% 完成率、95% 工具召回、平均 3.5 步 / 26s。
-   **早报系统 (Morning Brief)**: 定时生成每日舆情早报，支持调试模式（间隔触发），前端自动轮询展示。
-   **异动告警 (Alert)**: 实时扫描高热事件，三级告警（critical/warning/info），前端自动弹出通知。
-   **PDF 导出**: 文章、事件、早报、AI 对话均可一键导出 PDF 报告（fpdf2 + 中文字体 / html2canvas 对比看板）。
-   **两级情感分析**: 请求链路内走关键词规则引擎（毫秒级，响应级兜底不写库），后台走单线程 BERT 队列（`Johnson8187/Chinese-Emotion` 8 维）精细补全并持久化，保证首次打开事件即有分布图、且永远不会把 CPU 打满。

## 2. 核心技术实现

### 2.1 全域深度采集引擎
已稳定接入 **8 个** 信源，通过 APScheduler 定时调度（默认 120s 间隔），24/7 不间断同步：
-   **综合门户**: 微博热搜、百度热搜、今日头条、知乎热榜。
-   **专业财经**: 财联社电报、华尔街见闻、澎湃新闻热榜。
-   **视频态势**: 哔哩哔哩全站实时热榜。

采集细节：
-   **正文抽取**: 通过 Jina Reader 将目标页面转为 Markdown 正文，针对头条、B站、知乎等平台有专用解析逻辑。
-   **Cookie 凭据**: 微博、百度、头条、B站、知乎 5 个平台需要 Cookie 认证，后 3 个为公开 API。
-   **共享连接池**: 所有爬虫共享 httpx AsyncClient，线程池隔离 DB 写操作。

### 2.2 事件聚合与可视化
-   **基础聚类**: 基于 Canonical Title（正则去噪后的标题）+ Jaccard 分词相似度，合并阈值 0.22，归一化标题一致加 0.42 强制合并。
-   **语义聚类（正式接入）**: 通过 `SEMANTIC_CLUSTER=1` 启用 Sentence-BERT (`BAAI/bge-small-zh-v1.5`) + FAISS IVF-Flat 向量聚类，复合相似度 `0.6*cosine + 0.25*time_decay + 0.15*platform_diversity`，阈值自适应（Otsu 双层 + P95 分位数），簇内复核确保合并质量。实现于 `app/services/semantic_cluster.py`。
-   **影响指数 (heat_score) 计算**:
    事件级影响指数由三项加总组成：
    1.  **文章热度均值**（0-300）：对事件内所有文章按单篇评分排序，取 Top-3 均值。单篇评分 = 排名奖励 `max(0, 120-rank)` + 原始热度奖励 `min(hot_value/10000, 160)` + 来源奖励（微博/知乎额外 +20）。
    2.  **跨平台奖励**（0-150）：每多覆盖一个平台 +30，上限 150。体现事件的跨域传播力。
    3.  **数量奖励**：`log₂(article_count) × 15`，体现信息量规模的边际递减增益。
    
    综合公式：`heat_score = avg(top3_article_scores) + min((platform_count-1)×30, 150) + log₂(article_count)×15`
-   **事件态势分析**: 利用 **ECharts 6** 深度定制的可视化模态框，提供：
    -   **热度指数**: 时间序列下的态势演化趋势图。
    -   **密度谱系**: 时间轴上的情报发布分布。
    -   **信源权重**: 跨平台的声量占比饼图。
    -   **关键词图谱**: **关键词完全由后端 `_extract_display_keywords` 产出**（jieba + TF-IDF + blocklist + 停用词 + 片段过滤），前端仅做展示排序，杜绝"标点硬切"引入的噪声词（如"第7个行政区""如今以"）。
    -   **情感雷达**: 8 维情绪分析环形图，首次加载走规则兜底保证非空分布，BERT 精标由后台单线程队列逐步落库。

### 2.3 独立语义索引实验流
-   **嵌入模型**: `BAAI/bge-small-zh-v1.5`，通过 `app/services/embedding.py` 按需增量生成文章向量。
-   **向量存储**: SQLite `article_embeddings` 表持久化 float32 向量，避免重复编码。
-   **索引结构**: 新增 `app/services/semantic_index.py`，构建内存 **FAISS IVF-Flat** 索引；该链路现已正式接入事件聚合主流程，并保留独立调试入口。
-   **召回与精排**: 先用 FAISS 召回 top-K=20 近邻，再按 `0.6 * cosine + 0.25 * time_decay + 0.15 * platform_diversity` 计算复合相似度。
-   **阈值自适应**: 基于当前候选边分数分布计算 P95 阈值，用于语义事件聚类主链与聚类预览。

### 2.4 交互体验
-   **搜索翻页**: 采用 **9 宫格布局**，兼顾视觉稳定性与信息承载量。
-   **中断控制**: 集成 `AbortController`，自动打断过期网络请求，消除 UI 竞态。
-   **混合检索**: **MeiliSearch 全文检索** + 后端 **12 分制逻辑重排**（标题全词 +12、关键词 +8、正文 +5）。
-   **统一搜索**: 一次查询同时返回事件、文章、时间轴线、平台分布。
-   **AI 对话**: 支持单篇文章上下文对话和通用 AI 研判对话。

### 2.5 Tool-Calling Agent · 自主规划研判
为区别于纯 RAG 的"检索+拼接"范式，项目在 `app/services/agent/` 构建了一套**轻量自研 Agent 引擎**（总代码 ~270 行，仅依赖 `langchain-core + langchain-openai`，不引入 LangGraph / AutoGen）。

-   **工具注册表（9 个原子能力）**: `ToolSpec` + `ToolRegistry` 把每个 Python handler 自动编译成 OpenAI `function` schema。当前注册：
    -   `search_events` · 关键词 + 时间窗口检索事件
    -   `get_event_detail` · 事件详情 + Top-K 关联文章
    -   `compare_events` · 多事件并列指标对比（热度 / 情绪 / 平台分布）
    -   `analyze_event_sentiment` · 按时间桶聚合 BERT 情绪序列
    -   `search_articles` · 关键词文章搜索
    -   `semantic_search_articles` · FAISS 向量语义搜索
    -   `list_hot_platforms` · 多平台热榜快照
    -   `get_morning_brief` · 当日早报内容直取
    -   `rank_events_by_sentiment` · 按情绪占比排序 Top-K 事件（聚合工具，一步拿结果）
-   **Agent Loop**: `AgentLoop.run()` 同步执行 **观察-思考-行动** 循环，单文件 `loop.py`：
    1. 构造 system prompt + user query + 前轮 observations
    2. 调 DeepSeek（流式输出，每个 token 实时推送 `llm_thinking_delta`）拿 `tool_calls` 或 `final content`
    3. 有 `tool_calls` → 多工具 **并发执行**（ThreadPoolExecutor，单工具时跳过线程池），把每个结果 append 为 `tool` 消息
    4. 无 `tool_calls` → 当前 content 即 final，循环结束
    5. 硬上限 `MAX_STEPS=8` / `MAX_CONSECUTIVE_ERRORS=2`，兜底防止空转
-   **SSE 流式**: `POST /api/agent/chat` 走 `text/event-stream`，实时推送 `llm_thinking` / `llm_thinking_delta` / `llm_done` / `tool_call` / `tool_result` / `final_delta` / `final` / `done` 事件；前端 `AgentTrace.vue` 按时间线渲染调用链，**LLM 思考过程流式展示**，最终答案以 **打字机逐字** 效果推进。每个步骤显示“思考 X.Xs · 执行 XXms”精确耗时。
-   **引用可溯源**: system prompt 要求输出必须 `event#<id>` / `article#<id>` 绑定 id；前端正则识别中英文两种写法（含"事件#N"），点击即打开对应 EventModal / NewsCard 原文验证，**形成"推理→证据"闭环**。
-   **评测（详见 `runtime/eval/agent_eval.md`）**:
    | 指标 | 20 query 结果 |
    |:---|:---|
    | 完成率 | 100% (20/20) |
    | 工具召回 | 95% |
    | 平均步数 | 3.5 (P95=6) |
    | 平均延迟 | 26 s (P95=42 s) |
    | 无幻觉引用 | 人工抽查 T1/C1 全部真实存在 |
-   **为什么不直接用 LangGraph**: 工程复杂度 ↑、依赖 ↑、黑盒 ↑；我们这套核心 loop 有 **11 个单测** 全部离线（FakeLLM 注入），切换 provider 只改 `llm_adapter.py` 一个文件，可审计可回滚。

### 2.6 AI 对话上下文与模式融合

-   **统一 Agent 模式**: 前端已移除普通/智能体模式切换，所有对话请求统一走 Tool-Calling Agent 链路，确保用户体验一致。
-   **追问建议**: Agent 完成研判后，前端基于本次调用的工具类型自动生成 2-3 个上下文相关的追问建议，用户点击即可深入。
-   **异常容错**: Agent 达到最大步数限制（MAX_STEPS=8）时自动给出 fallback 结论；对无意义输入（如纯数字、空消息），system prompt 已内置拒绝指引；前端在 Agent 未能给出结论时显示友好提示。

## 5. 版本迭代（CHANGELOG）

### v1.1 — W5 性能优化 + UI 整合 + Bug 修复
-   **feat(agent/streaming)**: LLM 调用从同步 `invoke` 改为流式 `stream`，思考过程实时逐 token 推送 `llm_thinking_delta` 事件，前端 `AgentTrace.vue` 即时展示 LLM 推理文字流。
-   **feat(agent/timing)**: 每步显示精确的"思考 X.Xs · 执行 XXms"双段耗时（`llm_done` 事件含 `llm_latency_ms`），替代原来只显示工具执行时间的不准确标注。
-   **feat(agent/concurrency)**: 同一步多 tool_calls 改为 `ThreadPoolExecutor` 并发执行（最多 4 线程），单工具时跳过线程池；Agent 评测延迟从 37s 降至 26s。
-   **feat(agent/tool)**: 新增第 9 个工具 `rank_events_by_sentiment`（按情绪占比对事件排序 Top-K），聚合一步出结果，避免逐事件串行查询。
