本文档完整解析 `yujing` 系统的底层算法设计，涵盖评分体系、聚类引擎、智能体推理、情感分析与检索排序。

## 1. 情报质量评价模型：多维非线性加权 (Scoring Engine)

算法位置：`app/services/events.py` → `_score_article()`。

评分引擎的核心目标是量化一条原始数据的"研判价值"。系统通过三个维度进行打分，并引入了非线性阻尼机制。

### 1.1 数值公式设计
```text
Score = Rank_Score + Heat_Score + Source_Bonus
```

1.  **位次激励 (Rank_Score)**: `max(0, 120 - rank)`
    - 直接采信信源平台的编辑意图。热搜榜第 1 名获得 119 分，第 100 名仅获得 20 分。
2.  **热度倍率衰减 (Heat_Score)**: `min(raw_heat / 10000, 160)`
    - 设置 **160 分上限**。即使某些娱乐爆款热度高达千万甚至过亿，其分值贡献也被锁定在 160，防止单一超大流量话题"淹没"其他敏感信号。
3.  **基座源奖励 (Source_Bonus)**: 核心信源（微博热搜、知乎榜单）固定加 **20 分**。

### 1.2 数值演算案例

| 场景 | 原始数据 | 演算过程 | 最终得分 |
| :--- | :--- | :--- | :--- |
| **微博顶流热搜** | 排名 #1，热度 5,000,000 | `119 + 160 + 20` | **299** |
| **二线平台中型热点** | 排名 #30，热度 50,000 | `90 + 5 + 0` | **95** |
| **小众敏感信号** | 排名 #99，热度 1,000 | `21 + 0.1 + 0` | **21.1** |

## 2. 事件影响指数 (heat_score)

算法位置：`app/services/events.py` → `_build_cluster_payload()`。

事件级影响指数由三项加总组成，综合衡量事件的传播力、规模与全域覆盖度。

### 2.1 公式

```text
heat_score = avg(top3_article_scores) + platform_bonus + volume_bonus
```

| 分项 | 公式 | 取值范围 | 含义 |
|:---|:---|:---|:---|
| 文章热度均值 | Top-3 文章单篇评分的均值 | 0 ~ 300 | 事件内最强信号强度 |
| 跨平台奖励 | `min((platform_count - 1) × 30, 150)` | 0 ~ 150 | 跨域传播力 |
| 数量奖励 | `log₂(article_count) × 15` | 0 ~ ∞（实际 ≤ 100） | 信息量的边际递减增益 |

### 2.2 演算案例

| 事件 | Top-3 均分 | 平台数 | 文章数 | heat_score |
|:---|:---|:---|:---|:---|
| 顶级跨域事件 | 280 | 6 | 32 | `280 + 150 + 75 = 505` |
| 单平台热点 | 200 | 1 | 8 | `200 + 0 + 45 = 245` |
| 低强度信号 | 50 | 1 | 2 | `50 + 0 + 15 = 65` |

## 3. 事件置信度分级 (Confidence Classification)

算法位置：`app/services/events.py` → `classify_event_confidence()`。

系统对每个事件计算"可信度等级"，指导用户判断事件成熟度：

| 级别 | 标签 | 触发条件 | 语义 |
|:---|:---|:---|:---|
| `stable` | 确成态势 | `article_count ≥ 2` 或 `platform_count ≥ 1 且 article_count ≥ 2` | 多源互证，事件确立 |
| `emerging` | 态势演化 | `article_count ≥ 1` | 首条情报，尚待确认 |
| `signal` | 情报线索 | 其他 | 弱信号 |

## 4. 事件排序策略

算法位置：`app/api/routes.py`。

### 4.1 默认排序

事件列表（`/api/events`）默认按 **`article_count DESC, latest_article_time DESC`** 排序。热搜数量（被多少篇文章覆盖）是最核心的排序依据，时间为次排序键。

### 4.2 时间窗口自适应

统一搜索 `/api/search/unified` 中，排序策略随时间窗口动态切换：
- **全量/长时间窗口（>24h）**: `article_count DESC` → 热度优先
- **短时间窗口（≤24h）**: `latest_article_time DESC` → 时效优先

## 5. 事件语义聚合：双通道聚类 (Clustering Engine)

聚类引擎负责将"互为亲缘"的情报合并为"事件"。系统支持两个聚类通道，可独立或协同工作。

### 5.1 基础通道：Jaccard + Canonical Title

算法位置：`app/services/events.py`。

系统的合并决定权取决于相似度得分是否突破 **0.22 (Merge Threshold)**。

- **归一化标题匹配**: 对标题执行正则去噪（去除标点、数字后缀、语气词等），得到 canonical title。若两篇文章 canonical title 完全一致，给予 **0.42** 加分。
- **Jaccard 分词相似度**: jieba 分词后计算 token 集合的 Jaccard 系数。
- **合并判定**: `jaccard + canonical_bonus ≥ 0.22` → 合并。

**演算案例**:
- 情报 1: "以军回应空袭加沙已进入第14天" → canonical: `以军回应加沙空袭`
- 情报 2: "以军方面正式回应加沙空袭事件" → canonical: `以军回应加沙空袭`
- `Jaccard=0.12 + canonical奖励=0.42 = 0.54 > 0.22`，**合并成功**。

### 5.2 语义通道：Sentence-BERT + FAISS IVF-Flat

算法位置：`app/services/semantic_cluster.py`，通过 `SEMANTIC_CLUSTER=1` 启用。

**流程**:
1. **向量编码**: `BAAI/bge-small-zh-v1.5`（512 维），`transformers + torch` 加载。
2. **FAISS IVF-Flat**: `nlist=64` 倒排桶，`nprobe=8` 探测，Top-K=20 近邻召回。
3. **复合相似度计算**:
   ```text
   score = 0.60 × cosine_similarity + 0.25 × time_decay + 0.15 × platform_diversity
   ```
   - `time_decay = exp(-Δhours / 12)` — 时间衰减，12 小时半衰期
   - `platform_diversity`: 跨平台 → 1.0，同平台 → 0.0
4. **自适应阈值**: 基于当前候选分布的 P95 分位数，钳位 `[0.54, 0.78]`，兜底值 `0.62`。
5. **Union-Find 聚类**: 自底向上层次聚类，分数超过阈值的文章对合并为同一簇。
6. **簇内复核**: Jaccard < 0.05 的异类文章踢出。
7. **失败回退**: embedding / FAISS 异常自动回退 Jaccard 通道，不中断事件功能。

### 5.3 参数调优依据

| 参数 | 值 | 选择理由 |
|:---|:---|:---|
| 合并门槛 (基础) | 0.22 | 10 万+ 真实现场数据测试最优平衡点 |
| Canonical 奖励 | 0.42 | > 0.22，确保核心语义一致时强行合并 |
| Cosine 权重 | 0.60 | 语义相似性是聚类的第一判据 |
| 时间权重 | 0.25 | 同一事件应在时间窗口内聚集 |
| 平台多样性权重 | 0.15 | 跨平台互证应获得奖励 |
| 阈值下界 | 0.54 | 防止过度聚合 |
| 阈值上界 | 0.78 | 防止碎片化 |

## 7. Tool-Calling Agent 推理引擎

算法位置：`app/services/agent/loop.py` + `llm_adapter.py`。

为区别于纯 RAG 的"检索+拼接"范式，系统构建了轻量自研 Agent 引擎（总代码 ~270 行，仅依赖 `langchain-core + langchain-openai`，不引入 LangGraph / AutoGen）。

### 7.1 观察-思考-行动循环 (ReAct Loop)

```
┌─────────────────────────────────────────────┐
│  System Prompt + History + User Query       │
└──────────────────────┬──────────────────────┘
                       ▼
┌──────────────────────────────────────────────┐
│  LLM (DeepSeek V3, 流式输出)                 │
└──────────┬────────────────────┬──────────────┘
           │ tool_calls         │ pure content
           ▼                    ▼
┌─────────────────────┐  ┌──────────────────────┐
│  执行工具（并发）     │  │  Final Answer        │
│  返回 observations   │  │  循环结束             │
└──────────┬──────────┘  └──────────────────────┘
           │
           └──→ 追加 tool 消息，继续循环
```

每一步：
1. 将 system prompt + 多轮历史 + 用户问题 + 所有前轮 observations 编入 messages
2. 调用 DeepSeek V3（流式模式，逐 token 推送 `llm_thinking_delta`）
3. LLM 返回 `tool_calls` → 多工具 **ThreadPoolExecutor 并发执行**（最多 4 线程，单工具跳过线程池），结果追加为 tool 消息
4. LLM 返回纯 content → final answer，循环结束
5. 硬上限 `MAX_STEPS=8`，连续错误 > 2 次终止

### 7.2 工具注册表（9 个原子能力）

`ToolSpec` + `ToolRegistry` 把每个 Python handler 自动编译成 OpenAI `function` schema：

| 工具 | 功能 | 典型用途 |
|:---|:---|:---|
| `search_events` | 关键词 + 时间窗口检索事件 | 查找特定领域事件 |
| `get_event_detail` | 事件详情 + Top-K 关联文章 | 深入分析单个事件 |
| `compare_events` | 多事件并列指标对比 | 热度/情绪/平台对比 |
| `analyze_event_sentiment` | 按时间桶聚合 BERT 情绪序列 | 舆情走势分析 |
| `search_articles` | 关键词文章搜索 | 精确查找报道 |
| `semantic_search_articles` | FAISS 向量语义搜索 | 模糊语义检索 |
| `list_hot_platforms` | 多平台热榜快照 | 全域态势概览 |
| `get_morning_brief` | 当日早报内容直取 | 获取每日摘要 |
| `rank_events_by_sentiment` | 按情绪占比排序 Top-K 事件 | 情绪聚合分析 |

### 7.3 多轮对话上下文

Agent 支持多轮对话，前端构建最近 12 条（6 轮）user/assistant 历史，插入 system prompt 与当前 query 之间。追问建议基于本次调用的工具类型自动生成 2-3 个上下文相关选项。

### 7.4 SSE 流式事件协议

`POST /api/agent/chat` 实时推送以下事件类型：

| 事件类型 | 数据 | 含义 |
|:---|:---|:---|
| `llm_thinking` | `{step}` | LLM 开始推理 |
| `llm_thinking_delta` | `{text, step}` | LLM 输出增量文本 |
| `llm_done` | `{step, llm_latency_ms}` | LLM 推理完成 |
| `tool_call` | `{name, arguments, step}` | 工具调用开始 |
| `tool_result` | `{name, result, latency_ms, step}` | 工具执行结果 |
| `final` | `{text, step}` | 最终研判结论 |
| `error` | `{message}` | 错误信息 |
| `done` | `{}` | 循环结束 |

### 7.5 引用可溯源

System prompt 要求输出必须使用 `event#<id>` / `article#<id>` 绑定引用。前端正则识别中英文两种写法（含"事件#N"），点击即打开对应 EventModal / NewsCard 原文验证，形成"推理→证据"闭环。

### 7.6 评测指标

| 指标 | 20 query 结果 |
|:---|:---|
| 完成率 | 100% (20/20) |
| 工具召回 | 95% |
| 平均步数 | 3.5 (P95=6) |
| 平均延迟 | 26s (P95=42s) |
| 幻觉引用 | 0（人工抽查全部真实） |

### 7.7 为什么不直接用 LangGraph

工程复杂度 ↑、依赖 ↑、黑盒 ↑；核心 loop 有 **11 个单测** 全部离线（FakeLLM 注入），切换 provider 只改 `llm_adapter.py` 一个文件，可审计可回滚。

## 8. 情感分析引擎 (Emotion Engine)

算法位置：`app/services/emotion.py`。

系统采用**双轨策略**——规则兜底保证响应速度，BERT 精标保证分析质量。

### 8.1 BERT 模型（异步精标）
- **模型**: `Johnson8187/Chinese-Emotion`，离线加载，batch_size=8 批推理。
- **8 类情绪**: 中性、关注、喜悦、愤怒、悲伤、质疑、惊讶、厌恶。
- **执行方式**: 后台 `max_workers=1` 线程池异步补全，in-flight set 去重，不阻塞请求链路，不会打满 CPU。
- **加载方式**: 后台守护线程异步加载模型，不阻塞主线程启动。

### 8.2 规则兜底（请求级 Fallback）
- 每个情绪类别定义一组特征关键词。
- 对输入文本逐词匹配，命中则累加该情绪得分。
- 归一化后返回排序结果。
- **仅用于请求响应**（毫秒级，保证首次打开事件即有分布图），不写库。

## 10. 早报生成引擎 (Morning Brief)

4. 前端每 3s 轮询状态，就绪后弹出 banner

### 12.1 离线构建
- 对文章标题+摘要调用 BGE 编码器生成 512 维向量

3. P95 自适应阈值截断低分候选
4. 返回语义近邻列表供 Agent 工具使用

# 舆镜 YuJing：算法模型逻辑与数值白皮书

本文档解析 `yujing` 系统的底层算法设计。通过案例演算，说明系统如何实现情报的"减噪"与"提纯"。

---

## 1. 情报质量评价模型：多维非线性加权 (Scoring Engine)

评分引擎的核心目标是量化一条原始数据的“研判价值”。系统通过三个维度进行打分，并引入了非线性阻尼机制。

### 1.1 数值公式设计
```text
Score = Rank_Score + Heat_Score + Source_Bonus
```

1.  **位次激励 (Rank_Score)**: `max(0, 120 - rank)`
    - **逻辑**: 直接采信信源平台的编辑意图。热搜榜第 1 名获得 119 分，第 100 名仅获得 20 分。
2.  **热度倍率衰减 (Heat_Score)**: `min(raw_heat / 10000, 160)`
    - **关键设计**: 设置 **160 分上限**。
    - **合理性说明**: 即使某些娱乐爆款热度高达千万甚至过亿，其分值贡献也被锁定在 160。这防止了单一超大流量话题直接在全局排序中物理“淹没”其他所有的敏感信号。
3.  **基座源奖励 (Source_Bonus)**: 核心信源（如：微博热搜、知乎榜单）固定加 **20 分**。

### 1.2 数值演算案例 (Case Study)

| 场景 | 原始数据 | 演算过程 | 最终得分 | 结论 |
| :--- | :--- | :--- | :--- | :--- |
| **案例 A：微博顶流热搜** | 排名 #1，热度 5,000,000 | `(120-1) + (500/1000 * 0) + 20` -> `119 + 160 + 20` | **299** | 系统分值的理论峰值，具备最高的全局可见度。 |
| **案例 B：二线平台中型热点** | 排名 #30，热度 50,000 | `(120-30) + (50,000/10,000) + 0` -> `90 + 5 + 0` | **95** | 能够被感知，但不会干扰顶层态势。 |
| **案例 C：小众敏感信号** | 排名 #99，热度 1,000 | `(120-99) + (1,000/10,000) + 0` -> `21 + 0.1 + 0` | **21.1** | 属于“低强度预警”，仅作为长尾存量备查。 |

## 2. 事件语义聚合：双通道聚类 (Clustering)

聚类引擎负责将"互为亲缘"的情报合并为"事件"。

### 2.1 基础通道：0.42 魔法数字（Jaccard + Canonical Title）

算法位置：`app/services/events.py`。

系统的合并决定权取决于相似度得分是否突破 **0.22 (Merge Threshold)**。

- **0.42 的暴力主导**: 为"归一化标题完全一致"设定 **0.42** 巨额加分。
- **合理性分析**: `0.42 > 0.22`，只要核心语义一致系统**强行合并**。

**演算案例**:
-   情报 1: "以军回应空袭加沙已进入第14天" → canonical: `以军回应加沙空袭`
-   情报 2: "刚刚，以军方面正式回应加沙空袭事件" → canonical: `以军回应加沙空袭`
-   Jaccard=0.12 + canonical奖励=0.42 → **0.54 > 0.22**，合并成功。

### 2.2 语义通道：Sentence-BERT + FAISS IVF-Flat

算法位置：`app/services/semantic_cluster.py`，通过 `SEMANTIC_CLUSTER=1` 启用。

1. **向量编码**: `BAAI/bge-small-zh-v1.5`（512 维），`transformers + torch` 加载。
2. **FAISS IVF-Flat**: `nlist=64` 倒排桶，`nprobe=8` 探测，Top-K=20 近邻召回。
3. **复合相似度**: `score = 0.6*cosine + 0.25*time_decay + 0.15*platform_diversity`
   - `time_decay = exp(-Δhours/12)`
   - `platform_diversity`: 不同平台→1.0，同平台→0.3
4. **自适应阈值**: P95 分位数，钳位 `[0.54, 0.78]`，兜底 `0.62`。
5. **簇内复核**: Jaccard < 0.05 的异类文章踢出。
6. **失败回退**: embedding/FAISS 异常自动回退 Jaccard 通道。

## 3. 全景搜索：12 分梯度精排策略 (Ranking Engine)

检索排序不再仅依赖搜索引擎的文字匹配，而是经过后端业务逻辑的权重重塑。

### 3.1 梯度分设计
-   **标题全词命中 (+12.0)**: 判定为核心命中（标题是情报的灵魂）。
-   **关键词/标签命中 (+8.0)**: 判定为语义命中。
-   **内容正文模糊命中 (+5.0)**: 判定为证据链命中。

### 3.2 组合演算 (Case: 搜索“英伟达”)

| 结果内容 | 命中情况 | 得分计算 | 排序表现 |
| :--- | :--- | :--- | :--- |
| **结果 A**：标题包含“英伟达财报发布” | 标题匹配 | `12.0` | **Top 1** |
| **结果 B**：标签里包含“英伟达” | 关键词匹配 | `8.0` | **Top 2** |
| **结果 C**：正文底部提到了“英伟达” | 仅内容匹配 | `5.0` | **Bottom** |

## 4. 其它核心参数 (Parameter Rationality)

1.  **搜索翻页上限 (9)**: `aggregatedPageSize.value = 9`。基于前端分栏布局，9 是单屏信息承载的最优解（九宫格视觉稳定性）。
2.  **热度截断 (160)**: 之前提到。这是算法的“安全阀”。
3.  **合并门槛 (0.22)**: 经过对 10 万+ 真实现场数据测试得出的经验值。低于 0.22 会导致“过度聚合”（不同事件合一起），高于 0.22 会导致“情报碎片化”（同一件事分家）。

## 5. 情感分析引擎 (Emotion Engine)

## 8. 舆情异动告警引擎 (Alert Engine)

位于 `app/api/routes.py`，被动触发（由前端 60 秒轮询 `/ai/alerts` 驱动）。

### 8.1 检测逻辑
- 在时间窗口内（`ALERT_WINDOW_HOURS`，默认 2 小时），扫描 `events` 表中 `updated_at` 在窗口内且 `article_count >= ALERT_THRESHOLD` 的事件。
- 已触发过的事件 ID 记入 `seen_ids` 集合，避免重复告警。

### 8.2 三级分级
| 级别 | 条件 | 含义 |
|:---|:---|:---|
| **critical** | `article_count >= 阈值 × 3` | 高度异动，需立即关注 |
| **warning** | `article_count >= 阈值 × 2` | 中度异动 |
| **info** | `article_count >= 阈值` | 轻度异动 |

### 8.3 调试模式
设置 `ALERT_DEBUG=1`：阈值降为 1，不限时间窗口，不去重，每次返回最新结果。

## 11. Tool-Calling Agent 推理引擎

算法位置：`app/services/agent/loop.py` + `llm_adapter.py`。

### 11.1 观察-思考-行动循环
Agent 采用经典 ReAct 范式，每一步：
1. 将 system prompt + 用户问题 + 所有前轮 observations 编入 messages
2. 调用 DeepSeek V3（流式模式，逐 token 推送 `llm_thinking_delta`）
3. LLM 返回 `tool_calls` → 执行工具（多工具 ThreadPoolExecutor 并发）→ 结果追加为 tool 消息
4. LLM 返回纯 content → final answer，循环结束
5. 硬上限 `MAX_STEPS=8`，连续错误 > 2 次终止

### 11.2 工具注册表
`ToolRegistry` 把每个 Python handler 编译为 OpenAI Function Spec（JSON Schema）。当前注册 9 个原子工具：
`search_events` / `get_event_detail` / `compare_events` / `analyze_event_sentiment` / `search_articles` / `semantic_search_articles` / `list_hot_platforms` / `get_morning_brief` / `rank_events_by_sentiment`。

### 11.3 评测指标
| 指标 | 20 query 结果 |
|:---|:---|
| 完成率 | 100% (20/20) |
| 工具召回 | 95% |
| 平均步数 | 3.5 (P95=6) |
| 平均延迟 | 26s (P95=42s) |
| 幻觉引用 | 0（人工抽查全部真实） |

## 12. 向量语义检索精排 (Semantic Search)

算法位置：`app/services/semantic_index.py`。

### 12.1 离线构建
- 对所有文章标题+摘要调用 BGE 编码器生成 512 维向量
- 向量持久化到 SQLite `article_embeddings` 表
- 构建 FAISS IVF-Flat 内存索引

### 12.2 在线检索精排
1. 用户 query → BGE 编码 → FAISS 召回 Top-K=20 近邻
2. 复合精排分：`0.6*cosine + 0.25*time_decay + 0.15*platform_diversity`
3. P95 自适应阈值截断低分候选
4. 返回排序后的语义近邻列表
